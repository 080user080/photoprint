# PhotoPrint — Статус та план розвитку

## Призначення

Програма для автоматичної (за потреби — ручної) підготовки неякісних фотографій та зображень документів до друку.

**Що обробляється:**
- Фото документів, зняті з перспективою, з поганим освітленням, розмиті або з низькою контрастністю
- Сканоподібні зображення, які потребують авто-корекції: перспектива, яскравість, контраст, різкість

**Що пропускається без обробки:**
- DOCX, XLSX, PDF та інші офісні файли — відправляються на принтер за замовчуванням (priPrinter) напряму

**Поточний режим роботи:**
- GUI-додаток з чергою файлів, прев'ю До/Після та двома режимами: Авто та Ручний
- Пакетна обробка зображень у фоновому потоці
- Відправка на друк через системний принтер (Windows: ShellExecute / mspaint, Linux/Mac: lp)

## Мета

Створити **віртуальний принтер**, який:
1. Отримує будь-який друкований документ із будь-якої програми
2. Автоматично визначає, чи потребує зображення обробки
3. Застосовує оптимальні корекції без участі користувача
4. Перенаправляє результат на фізичний принтер за замовчуванням (priPrinter)

**Кінцевий результат:** користувач натискає «Друк» у Word, браузері чи іншій програмі — а PhotoPrint у фоні вирішує, що робити, і видає готовий документ на друк.

## Пріоритети розвитку (актуальні)

### 0. Часткова корекція перспективи (ВИСОКИЙ ПРІОРИТЕТ)
- Вирівнювання тільки однієї сторони документа, не чіпаючи інші
- Визначення рівної сторони (горизонтальної чи вертикальної)
- Визначення кривої сторони через різницю координат
- Частковий warp тільки для кривої сторони
- Збереження інших сторін без змін

#### План реалізації:

**Крок 1: Визначення рівної сторони**
- Порівняти різницю Y для верху і низу (горизонтальність)
- Порівняти різницю X для лівої і правої сторони (вертикальність)
- Сторона з меншою різницею вважається рівною

**Крок 2: Визначення кривої сторони**
- Якщо top_diff > bottom_diff → кривий верх
- Якщо bottom_diff > top_diff → кривий низ
- Якщо left_diff > right_diff → крива ліва сторона
- Якщо right_diff > left_diff → крива права сторона

**Крок 3: Частковий warp для кривої сторони**
- Вирівнюємо тільки криву сторону (усереднюємо координати)
- Інші сторони залишаємо без змін
- Застосовуємо getPerspectiveTransform з новими точками

**Крок 4: Інтеграція в GUI**
- Додати перемикач: "Повна перспектива" / "Часткова перспектива"
- В ручному режимі — автоматично визначати криву сторону
- В авто режимі — застосовувати часткову корекцію за замовчуванням

**Крок 5: Валідація**
- Поріг різниці координат для визначення "кривої" сторони (наприклад, 10px)
- Перевірка на випадки коли всі сторони рівні (не застосовувати корекцію)
- Перевірка на випадки коли всі сторони криві (повна перспектива)

#### Очікуваний результат:
- Документ вирівнюється тільки там де це потрібно
- Не деформується частина документа яка вже рівна
- Менше артефактів при корекції перспективи

### 0.1 Захист оригіналу від змін (ВИСОКИЙ ПРІОРИТЕТ)
- Оригінальне зображення (_orig) не має змінюватися жодною функцією
- Прев'ю "До" (before preview) має завжди показувати оригінал без змін
- Всі корекції застосовуються тільки до _base та _processed
- Забезпечити immutability оригіналу на рівні коду

#### План перевірки всіх функцій:

**Крок 1: Аудит функцій в main_window.py**
- Перевірити всі методи які викликають `self._preview.set_before()`
- Перевірити чи не змінюється `self._orig` в жодному місці
- Перевірити чи не передається `_orig` в pipeline для модифікації

**Крок 2: Аудит функцій в pipeline.py**
- Перевірити чи всі функції повертають нове зображення, а не модифікують вхідне
- Перевірити чи немає in-place операцій які змінюють оригінал

**Крок 3: Аудит функцій в processing модулях**
- Перевірити всі функції обробки на immutability
- Додати .copy() там де це потрібно для захисту від змін

**Крок 4: Додати тести для immutability**
- Написати тести які перевіряють що вхідне зображення не змінюється
- Порівнювати вхідне зображення до і після виклику функції

**Крок 5: Додати захист на рівні коду**
- Розглянути використання @dataclass(frozen=True) або інших механізмів immutability
- Додати assert або логування якщо оригінал змінюється

#### Очікуваний результат:
- Оригінал завжди незмінний
- Прев'ю "До" завжди показує оригінал
- Користувач може будь-коли повернутися до оригіналу через скидання

### 1. Збереження максимальної якості
- Усі операції на оригінальному розмірі (без downsample до обробки)
- Прев'ю тільки для відображення в GUI, обробка на повному розмірі
- Оптимізація пам'яті для великих зображень (streaming по частинах)

### 2. Автоматичне визначення яскравості
- Аналіз гістограми для визначення недоливу/перелиття
- Авто-корекція до оптимального діапазону (5-95% перцентилів)

### 3. Автоматичне визначення контрасту
- Аналіз розподілу яскравості для виявлення низькоконтрастних зображень
- Авто-контрастна корекція з адаптивними порогами (5-95% перцентилів)

### 4. Автоматичне визначення різкості
- Аналіз частот (FFT або Laplacian variance) для виявлення розмитості
- Авто-різкість на основі виміряного blur-фактора

### 5. HDR для документів
- Поточний HDR працює добре для фото, але погано для документів
- Проблема: при переході світла додає чорного в світлих частинах → негативно для друку
- Потрібно:
  - Визначати тип зображення: документ (чорний текст на білому фоні) чи фото
  - Для ч-б документів: чорні букви на білому фоні (пріоритет)
  - Для кольорових документів на іншому фоні (грамоти, посвідчення, паспорти): адаптивна обробка
  - Для фото: використовувати поточний HDR

### 6. Визначення типу документа
- Розпізнавання: чорнобілий документ (телефон фотографує з шумами, не контрастно) чи документ на неоднотонному фоні
- Для ч-б документів: бінаризація, чорні букви на білому фоні (як стандартний офісний документ)
- Передбачити можливість кольорових документів (білі/червоні на іншому фоні)
- Чорне по білому — пріоритет

### 7. Всі налаштування в GUI
- Всі параметри (масштаби, пороги, коефіцієнти) мають бути доступні через GUI
- Немає потреби переписувати код для зміни налаштувань
- Можливість експорту/імпорту профілів налаштувань

### 8. Запам'ятовування ручних корекцій по файлу (середній пріорітет)
- У ручному режимі при переключенні між файлами черги — зберігати налаштування яскравості, контрасту, різкості, HDR, перспективи для кожного файлу окремо
- При поверненні до файлу — відновлювати його останні налаштування
- Скидання по кнопці «Скинути всі» — тільки для поточного файлу

### 9. Авто-детекція перспективи з fallback
- Автоматичне знаходження документа на фоні та виправлення перспективи
- Fallback для документів без чітких контурів (наприклад, блідий документ на блідому фоні)
- Можливість ручної корекції, якщо авто не спрацювало

## План покращення автоматичної перспективи

### Проблеми поточної реалізації

1. **AdaptiveThreshold з `BINARY_INV`** — працює тільки коли документ світліший за фон
2. **Немає fallback** — якщо не знайшло 4 кути, повертає `None`
3. **Немає resize перед аналізом** — великі зображення = повільно + шум
4. **Немає перевірки пропорцій** — може знайти "документ" 1:50 (лінію)
5. **Немає Canny як альтернативи** — для кольорових фонів Canny краще

### План реалізації

#### Крок 1: Resize для швидкості та стабільності
- Максимальний розмір для аналізу: 800px
- Масштабування координат назад після пошуку

#### Крок 2: Множинні методи детекції
- **Спроба 1**: Adaptive Threshold (для ч-б документів)
